Buttons

Button Types

Use as described. Remember that links, divs and spans are not buttons.

Standard

Usage: Used as a generic button. Should have simple text that describes the action.

Rules: May include an appropriate icon with text.

<button class="btn btn-default">Standard</button>

<button class="btn btn-default" disabled="disabled">Disabled</button>

<button class="btn btn-default"><span class="icon-add">Icon + Text</span></button>

Primary

Usage: Used when there’s a need for a suggested action, like the “Done” button in a modal or an obvious "next step".

Rules: There shouldn’t be more than one primary button in any dialog/modal/interface.

<button class="btn btn-primary">Primary</button>

<button class="btn btn-primary" disabled="disabled">Disabled</button>

Icon + Label

Usage: Used to indicate a supporting action, typically inside of a tab or accordion menu.

Rules: To avoid being confused with a standard link, these buttons must always be accompanied with an appropriate icon.

<button class="btn btn-text icon-info">Icon + Label</button>

<button class="btn btn-text icon-info" disabled="disabled">Disabled</button>

Special Action

Usage: Used in an application to denote a special action. It has a “go” connotation, so it should be used to initiate a journey of some sort (like “new project”, “new campaign”, “launch campaign”).

Rules: Maximum of one per application (although having none is fine).

<button class="btn btn-special">Special</button>

<button class="btn btn-special" disabled="disabled">Disabled</button>

Cancel

Usage: Used in any dialog box/modal/menu where actions require a commit button, there should always be an option to cancel as well. Cancel should back out any changes that occurred and leave it in the previous state.

Rules: Should always be placed to the left of a Primary of standard button.

<button class="btn btn-cancel">Cancel</button>

<button class="btn btn-cancel" disabled="disabled">Cancel</button>

Delete or Remove

Usage: A stand alone “x” icon (never the letter "x" or "X") used to remove or delete an item.

Rules: Typically placed on the far right side in table rows. It may be appropriate to have a confirmation alert if the action cannot be undone.

<button class="btn btn-remove"></button>

<button class="btn btn-remove" disabled="disabled"></button>

Icon Buttons

Usage: Only for commonly used and easy to understand actions in the main area of an application, otherwise use a standard or “icon + label” button with descriptive text.

Rules: Icon buttons must have descriptive tooltips that should appear on hover after a short delay.

<button class="btn btn-icon icon-sort-by"></button>

<button class="btn btn-icon icon-sort-by" disabled="disabled"></button>

<button class="btn btn-icon icon-sort-by active"></button>

Toggle Buttons

Usage: Used like a checkbox to filter viewport information.

<button class="btn btn-toggle" data-toggle="button">Toggle Button</button>

<button class="btn btn-toggle" disabled="disabled">Toggle Button</button>

Links

Usage: Links send the user away from the current page to somewhere else.

Rules:

  • Links should not look like buttons, they always display an underline on hover.
  • When a link’s destination is to another Beacon application, it may be accompanied with the icon-go-to for internal links (), otherwise a link should not have any associated icons.
  • The <a href=”#”> markup is prefered for links, but if for some reason you need to, you can style buttons as links using <btn-link> class

<button class="btn btn-link">Button Link</button>

<button class="btn btn-link" disabled="disabled">Disabled</button>

Button Sizing

The default size should be used in most cases, but these are also available when necessary.

<button class="btn btn-default btn-xs">Extra Small</button>

<button class="btn btn-default btn-sm">Small</button>

<button class="btn btn-default">Default</button>

<button class="btn btn-default btn-lg">Large</button>

<button class="btn btn-default btn-block">Block</button>

Tooltips

These definitions are for actual tooltips, styled and defined separately from tooltip menus/modals.

Instant Tooltip

A tooltip that appears on hover with no delay, containing minimal text. This type of tooltip should be used in instances where the interaction method isn’t immediately discernible, such as with icon buttons that don’t have supporting text.

This tooltip method requires strong visual contrast. Informing the user of the expected result from a potentially illusive interaction method is key to providing the intended efficiency benefits (and reducing possible frustrations).

Verbage

Because this tooltip appears instantly, very minimal verbiage should be used. The user may experience multiple tooltips appearing as they rove around the page, so 4 to 5 words at the most is acceptable. Ideally, 1 or 2 words should be used.

Placement and Animation

For this tooltip method to be successful, it needs to be informative yet unobtrusive. Since these appear instantly on hover, there is a potential that a user may trigger multiple instances of these as they rove around the interface. Tooltips should preferably appear over whitespace without covering other UI elements (especially other interactive ones), but should also have a certain amount of consistency. For example, three icon buttons next to each other should have tooltips appear in the same direction, not in three different ones.

Hover for tooltip:

Although this tooltip is meant to appear instantly, to avoid confusing the user a very fast 100ms fade in helps tell the user that the tooltip has appeared. <button class="btn btn-text btn-large icon-add bui-instant-tooltip" data-toggle="tooltip" data-placement="auto" title="Awesome Tooltip">Hover Here</button>

Code Note: Bootstrap powered tooltips must be initialized manually. To make things easier, add the class bui-instant-tooltip and the tooltip will be initialized automatically (as long as your document includes bui-scripts.js). When dynamically creating contant that needs a tooltip, just run buiTooltipInit(); to initialize any new tooltips.

Placement is set with the data-placement="" attribute. "auto" will make the tooltip appear on top unless it encounters a constraint in the viewport, at which point it will push to bottom, left, etc. The other available attributes are "top", "right", "bottom" or "left".

Delayed Tooltip

Being Developed

A tooltip that appears on hover after a short delay, possibly containing detailed information or even links to KB articles, etc. This type of tooltip should be used when an interaction method has some sort of label or cue to the expected outcome, but where more detailed information may be desirable. It can help to educate users in context.

Similar use of contrast as the instant tooltip, which will help differentiate informational tooltips from menu tooltips. A delayed tooltip contains a title (or main text), this may be the only content in the tooltip.

(EXAMPLE: static tooltip with just a title)

If the interaction method is complicated or unique enough, there may be a need to include further information in the tooltip. An optional sentence or two may be added below the title, containing links if need be. A link may also appear in the top right for an easily visible “learn more” opportunity.

(EXAMPLE: static tooltip with title, further description and learn more link)

Verbage

The tooltip title should be short and descriptive, preferably 1 or 2 words with a max of 4 or 5. Further information in paragraph form below the title should not be more than 1 or 2 sentences.

Placement and Animation

Because this tooltip has a delay before appearing, there is more forgiveness if it happens to cover other UI elements (it’s size may require it anyway). Therefore this tooltip should have more consistent placement, always appearing above the hovered element unless obstructed by the bounds of the viewport.

The tooltip should animate in after a 700ms delay, fading in for 200ms. When the mouse exits, the tooltip should immediately fade out for 100ms.

Static

A static tooltip not dependent on a hover action, usually appearing on an initial screen to guide a user toward an intended first action. This tooltip can easily be considered obstructive or distracting. It should only be used if user testing has demonstrated its usefulness.

Because this tooltip is static and meant to clearly direct users to a desired action, the text is larger and more prominent.

A Static Tooltip
<div class="tooltip-static right" style="top: 40%; left: 5%;"> <!-- Position is 'absolute' by default -->
	<div class="tooltip-arrow"></div>
	<div class="tooltip-inner">A Static Tooltip</div>
</div>

Verbage

Verbage should be as short as possible while clearly describing the intended action. It will likely include action cues like “Select an option to…”, or “Start by adding…”.

Placement and Animation

Since this tooltip is static, it absolutely must not cover any UI elements or text.

After a user has made any action on the screen, this tooltip should fade out for 150ms.

Modals

The markup for bui modals are very similar to standard bootstrap modals. The close 'x' button and cancel buttons have bui specific classes.

Modal title

One fine body…

<div class="modal fade" id="bui-lib-example-modal">
  <div class="modal-dialog">
    <div class="modal-content">
      <div class="modal-header">
        <button type="button" class="close" data-dismiss="modal"></button>
        <h4 class="modal-title">Modal title</h4>
      </div>
      <div class="modal-body">
        <p>One fine body…</p>
      </div>
      <div class="modal-footer">
        <button type="button" class="btn btn-cancel" data-dismiss="modal">Cancel</button>
        <button type="button" class="btn btn-primary">Save changes</button>
      </div>
    </div><!-- /.modal-content -->
  </div><!-- /.modal-dialog -->
</div><!-- /.modal -->

Header

Title + Optional Drop-Down Menu:

Usage: The title should be short and descriptive. This title may be displayed as a Drop-Down menu in situations where the modal features a number of categories (e.g. the project menu in the survey building app).

Help Resources

Usage: There may be a “Learn More” link or help icon right aligned in the header.

Close "x" Button

Usage: A stand alone icon-close (never the letter "x" or "X") used to close out of a modal, should always appear in the top right corner.

Rules: Equivalent to cancel, any changes made and not executed should be reverted to the previous state.

Content Area

Usage: Any associated content should be displayed in here.

Rules: Do not use panel or accordion menus inside modals.

Footer

Usage: The footer of a modal should contain any buttons related to submitting, canceling, or saving the content. There may also be checkboxes that appear for settings related to the modal (e.g. “Show this screen for new projects”).

Rules: The primary button should always be the furthest right aligned, with secondary buttons to the left of that, and a cancel button to the left of all buttons. Any checkboxes in the footer should be aligned to the left.

Placement

The standard modal is placed 30px from the top and will allow scrolling if the modals content is long (or if the user is on a device with a small viewport).

If your modal has very little content and you're sure the user won't need the ability to scroll, you can apply the custom BUI modal-centered class to .modal, and center the modal in the window.

Typography

Most of this uses the same markup as bootstrap, but it's worth reading through as it also provides some best practice tips.

Font

The base font-size is 14px. All other font-sizes should be set using em or rem units so that the relative font size can be adjusted for different viewports.

The font weights available are 400 (+ italic), 600 (semi-bold), and 700.

The font family used for Beacon is Open Sans.

All HTML headings, <h1> through <h6> are styled and available.

All headers and titles should be written using title casing. As there are a few different ways of applying title casing, you can use this handy tool to make sure you’ve got it right. The rules we use for title casing come from the Chicago Manual of Style.

Emphasis

You can use HTML's default emphasis tags.

Small

Use the <small> tag for de-emphasizing text.

This might be useful for fine print text.

<small> . . . </small>

Bold

Use the <strong> tag for bold emphasis.

Making text bold is great for emphasis. Just remember, if too much text is bold, no text is bold.

<strong> . . . </strong>

Italic

Use the <em> tag for italic emphasis.

Italic text is good for emphasising a single word, or a whole phrase.

<em> . . . </em>

All Caps

Add class="caps" for all caps.

All caps may be good for certain section titles - they're loud, so use sparingly

<h4 class="caps"> . . . </h4>

Underline

Never use underline. Links underline themselves on hover, that's it.

Labels

Style labels by adding a class=”label”. Labels can be used for things like the survey state (dev, test, live, closed).
